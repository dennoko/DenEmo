# マルチフレームモード 機能比較・UX 評価・パフォーマンス調査レポート

作成日: 2026-06-11
最終更新: 2026-06-12（セクション 2・3 の改善案を実装。実装状況は末尾「5. 実装状況」を参照）
対象: DenEmo マルチフレーム（Animation）モード
調査範囲: `AnimationModeUI` / `AnimationClipModel` / `AnimationClipEditor` / `AnimationPreviewController` / `AnimationPlayback` / `AnimationTimelineUI.*` / `ShapeKeyListUI.*` / `DenEmoTimelineWindow`

## 1. 他ツールと比較した不足機能・改善候補

Unity 標準の Animation ウィンドウ、Timeline、および VRChat 界隈でよく使われるアニメーション編集ツール（VeryAnimation、FaceEmo 等）との比較。

### 1.1 不足している機能（優先度順）

#### 高優先度

| 機能 | 現状 | 補足 |
|---|---|---|
| **複数キーの選択・一括移動** | キーは 1 個（または 1 フレーム列）単位でしかドラッグできない | Unity 標準は矩形選択 → まとめて移動/削除/コピーが可能。表情アニメでは「後半全部を 5 フレーム後ろへずらす」操作が頻出するため、影響が大きい |
| **リタイミング（時間スケール）** | クリップ長や FPS を変えてもキー時刻は固定 | `AnimationClipEditor.SetFrameRate()` は `frameRate` を書き換えるだけでキーの時刻（秒）は維持されるため、FPS を変えるとフレーム番号上の位置がすべてズレる。Unity 標準はダイアログなしで同挙動だが、DenEmo は「フレーム」中心の UI なので体感のズレが大きい。FPS 変更時に「キーをフレーム位置基準で再配置するか」を選ばせるべき |
| **複数メッシュ（追加メッシュ）のアニメーション編集** | `AnimationClipModel.SmrPath` がメインメッシュ 1 本のパスでフィルタするため、追加メッシュ（眉・歯など）のカーブは表示も編集もできない | ポーズモードは複数メッシュ対応済みなので、モード間で機能の非対称がある。表情アニメは Body + Brow の組み合わせが定番なので実用上の穴になっている |
| **範囲コピー/ペースト** | コピーは「1 フレーム分のキー一式」のみ（`_keyClipboard`） | 区間（複数フレーム）のコピー、別トラックへのペーストができない |
| **カーブの可視化** | トラック行は ◆ の位置のみ。キー間の値の変化（カーブ形状）が見えない | フル機能のカーブエディタは過剰でも、トラック行の背景に値の折れ線をうっすら描くだけで「どこで急変しているか」が掴め、Ease/Linear の違いも見える |

#### 中優先度

- **キー値の直接編集**: キーの値を変えるには「シークしてスライダーを動かす」しかない。キーをダブルクリック（または右クリックメニュー）で数値入力できると速い。ホバーチップ（`F:12 V:80.0`）で値表示はあるので、編集導線だけが欠けている。
- **フレームの挿入/削除（リップル編集）**: 「この位置に 5 フレーム空きを挿入して以降を全部ずらす」「このフレームを詰めて削除」ができない。
- **再生範囲（ワークエリア）指定**: ループ再生は常にクリップ全体。「この 0.5 秒だけループで確認したい」ができない。口パクや瞬きの調整では区間ループが標準的なワークフロー。
- **ペースト時のミラー/反転**: L/R シェイプの値を入れ替えてペーストする機能。左右同期表示（ポーズモード由来）があるツールなので親和性が高い。
- **キー値 0〜100 制限の明示**: スライダーが 0〜100 固定。Unity 標準はオーバーシュート値も打てる。意図的な制約なら問題ないが、既存クリップに範囲外の値があった場合の挙動（表示・編集）が未定義。
- **未保存変更の警告**: クリップ差し替え・ウィンドウクローズ時に編集中クリップの保存確認がない（編集は `AnimationClip` インスタンスに直接行われるため Unity 終了時までは残るが、ユーザーには見えにくい）。

#### 低優先度（あると差別化になる）

- キー削減（カーブ単純化）: REC でスライダーを動かしながら録ると将来的に密なキーができるため、間引き機能。
- `F` キーで全体表示 / 選択キーへズームフィット（Unity / Blender 標準のショートカット）。

### 1.2 すでに優れている点（維持すべき）

- ボーンを触らないブレンドシェイプ専用プレビュー（`AnimationMode` 非使用）は VRChat アバター用途で正しい設計。
- フレーム/秒の 2 段ルーラー、ホバーチップ、ワークフローガイド、REC バナーなど初学者向け配慮。
- タイムラインの分離ウィンドウ化。
- スムーズループ（保存時のみ実体化）はクリップを汚さない良い設計。

## 2. マルチフレーム操作の直観性評価と改善案

### 2.1 値変更がクリップに記録される条件が暗黙的（最重要）

`AnimationModeUI.BuildDrawContext()` の `OnValueChanged` は次の 3 状態に分岐する:

1. REC 中 → キーを記録
2. REC オフでも、現在時刻に既存キーがある → **キーを上書き**
3. REC オフかつキーなし → SMR ウェイトに直接書くだけ（**シークすると消える**）

問題点:

- 状態 2 は「REC していないのにクリップが書き換わる」ため、試しにスライダーを触っただけでデータが変わる。Unity 標準の Animation ウィンドウも録画中は同様だが、録画オフ時にカーブを書き換えることはない。
- 状態 3 は逆に「調整したのにシークした瞬間に値が戻る」サイレントな作業ロス。ユーザーから見ると「スライダーが勝手に戻った」。

**改善案**:
- 状態 2 を REC 時のみに限定するか、少なくとも行のスライダー色・◆ の色で「いまの操作はクリップに書かれる/書かれない」を常時表示する。
- 状態 3 で値が記録されないままシークしようとしたとき、ステータスバーに「未記録の変更は破棄されます（◆ で記録）」を 1 回出す。

### 2.2 「全キー補間」トグルが破壊的

`DrawPlaybackControls` の Step/Linear/Ease トルバーは、(a) 新規キーのデフォルト補間 (`CurrentInterp`) の選択と、(b) `SetAllKeysInterpolation()` による**全既存キーの即時一括変更**を 1 つの UI で兼ねている。キーごとに補間を作り込んだ後にデフォルトを変えたくてクリックすると、全キーが上書きされる（Undo は効くが気づきにくい）。

**改善案**: デフォルト補間の選択と「全キーに適用」ボタンを分離する。一括適用は確認ダイアログ付きの明示的なボタンにする。

### 2.3 シーク操作のヒット領域が狭い

`HandleScrubberInput` はスクラバー帯（高さ 14px）のみが対象で、ルーラー（38px）をクリック/ドラッグしてもシークしない。Unity 標準・ほぼ全ての動画/アニメツールはルーラー上のドラッグでシークできるため、新規ユーザーは高確率でルーラーを掴もうとして空振りする。

**改善案**: ルーラー全域＋スクラバー帯をシーク対象にする（ズームコントロールのボタン領域は除外）。

### 2.4 ◆+（全シェイプにキー挿入）の対象が「表示中」依存

`DrawKeyframeActionRow` の ◆+ は `IsVisible` なシェイプにキーを打つため、検索ボックスやフィルタの状態で結果が変わる。フィルタ中であることを忘れていると「キーを打ったつもりのシェイプが抜ける」。逆に全シェイプ対象だと不要キーが大量に入る。現仕様は妥当だが暗黙的。

**改善案**: ボタンのツールチップ/確認に「表示中の N 個のシェイプに挿入」と件数を出す。トラックフィルタ有効時はトラック済みシェイプのみを対象にする選択肢も検討。

### 2.5 キードラッグのブロッキング挙動

`MoveKeys()` は隣のキーの 1 フレーム手前で停止する仕様。事故防止としては合理的だが、Unity 標準（重なったら上書き）と違うため「途中で引っかかって動かない」と感じる。現在はドラッグ中のフレーム番号表示があるので状況は分かるが、停止理由は表示されない。

**改善案**: ブロックされた瞬間に停止位置の境界を一瞬ハイライトする

### 2.6 マウスホイールの競合

タイムラインはメインウィンドウのスクロールビュー内にあり、トラック領域上ではホイールがズームに使われる（`HandleTimelineZoom` が `e.Use()`）。ウィンドウをスクロールしているつもりが、カーソルがタイムラインに差し掛かった瞬間ズームに化ける。

**改善案**: ズームを Ctrl+ホイールに割り当て、素のホイールはスクロールに流す（Unity の Animation ウィンドウも Ctrl+ホイール基準ではないが、埋め込み配置の DenEmo では修飾キー必須のほうが安全）。
分離ウィンドウ時のみ素のホイールズームを許可する手もある。
両方実装しつ？ボタンをセクションに一つ追加し、？のマウスオーバー時のツールチップで知らせたい。

### 2.7 キーボードショートカットの発見性と一貫性

- ショートカット（Space / ← → / , . / Del / Ctrl+C/V）の一覧が UI に出てこない。ツールチップかヘルプポップアップで提示したい。
- Delete は「現在フレームの**全トラック**のキー削除」で、1 トラックだけ消したい場合は右クリックメニューが必要。削除前の確認がない（Undo はあるが、◆ が複数消えても気づきにくい）。削除件数をステータスバーに出すと安心感が出る。
- リスト側のスライダーにフォーカスがあると矢印キーがタイムラインに渡らない等、フォーカス依存の挙動はあるが `GUI.GetNameOfFocusedControl()` ガードで概ね妥当。

### 2.8 その他の小改善

- **クリップ長の短縮時**: 範囲外に取り残されるキーの存在が UI から見えない。短縮時に「範囲外に N 個のキーがあります」と警告し、削除/維持を選ばせる。
- **トラック行の「＝」ハンドル**: フレーム列全体の移動ハンドルとして機能が伝わりにくい。ツールチップはあるが、アイコンを「↔」系にする、ホバー時に縦のガイドラインを出す等。
- **REC の押し忘れ対策**: 現状の赤バナーは良いが、REC オフでキーのない時刻でスライダーを動かした時こそ警告が必要（2.1 と同件）。

## 3. パフォーマンス問題（フリーズ級の重さ）の想定原因

報告症状「マルチフレームの処理が異様に重い（フリーズレベル）」について、(A) DenEmo 内部のホットパス、(B) 他ツールとの競合、の両面で洗い出した。

### 3.1 内部要因 — 確度が高い順

#### A-1. スライダードラッグ中のキー記録がイベント毎にフルコストで走る【最有力】

REC 中（または既存キー上）のスライダードラッグでは、**マウスドラッグイベント 1 回ごと**に以下が全部実行される（[AnimationModeUI.cs:213-228](UI/AnimationModeUI.cs#L213-L228) → [AnimationClipEditor.cs:38-50](Core/AnimationClipEditor.cs#L38-L50)）:

1. `Undo.RecordObject(Clip)` — **クリップ全体のシリアライズ**。トラック数 × キー数に比例。
2. `AnimationUtility.GetEditorCurve` + `SetEditorCurve` — `SetEditorCurve` はクリップ内部データの再構築を伴い、クリップ全体のカーブ数に比例して重い。
3. `Commit()` → `MarkDirty()` でトラックキャッシュ全無効化。
4. `Preview.SampleAt()` → `EnsureCache()` が**全トラックを `GetEditorCurve` で再取得**（[AnimationClipModel.cs:125-171](Models/AnimationClipModel.cs#L125-L171)）。トラック 100 本 × 各キー配列の再アロケートで GC 圧も高い。
5. 全トラック分 `SetBlendShapeWeight` + `SyncValuesFromMesh()`（全アイテムの `GetBlendShapeWeight`）。
6. `SceneView.RepaintAll()` — シーンビューのフル再描画。重いアバターシェーダー（Poiyomi/lilToon 等）では 1 回が数十 ms になり得る。

トラック数が多いアバター（表情シェイプ 150 個超は珍しくない）では 1 イベントあたり数十〜数百 ms × ドラッグ中は毎秒数十イベント → **イベント処理が滞留して「フリーズ」と体感される**。これは件数依存なので「一部のユーザーだけ異様に重い」という報告と整合する。

なお、**ポーズモードには 50ms スロットリング（`QueuePendingApply`）があるのに、アニメーションモードの `animContext` 経由のパスはスロットルを完全にバイパスしている**（[ShapeKeyListUI.Rows.cs:167-184](UI/ShapeKeyListUI.Rows.cs#L167-L184)）。モード間の実装非対称がそのまま重さの差になっている。

**対策案**:
- ドラッグ中は `RecordKey` をポーズモード同様に時間スロットル（50ms）し、マウスアップ時に最終値で確定する。
- `Undo.RecordObject` はドラッグ開始時（hotControl 取得時）に 1 回だけ呼ぶ。
- `MarkDirty()` の粒度を下げ、単一トラックの値変更ではそのトラックの `Curve`/`KeyTimes` だけ更新する部分無効化 API を `AnimationClipModel` に追加する。
- `SampleAt` の `SceneView.RepaintAll()` を `SceneView.lastActiveSceneView?.Repaint()` に絞る、またはスロットルする。
- `SyncValuesFromMesh` はトラック対象のシェイプのみ同期する。

#### A-2. 分離タイムラインウィンドウの常時 Repaint ループ

`DenEmoTimelineWindow.OnEnable` で `EditorApplication.update += Repaint`（[DenEmoTimelineWindow.cs:51-54](UI/DenEmoTimelineWindow.cs#L51-L59)）。これは**再生中でなくてもエディタ更新毎にウィンドウ全体を再描画し続ける**。Repaint 要求はエディタのアップデートループを最大速度で回し続けるため、ウィンドウを開いているだけで CPU/GPU を常時消費し、他の重さ（A-1 等）を増幅する。

**対策案**: 購読をやめ、再生中（`Playback.IsPlaying`）とインタラクション中のみ `Repaint()` を要求する。再生中も `Tick()` 側が既に `window.Repaint()` を呼んでいるため、この購読自体が不要の可能性が高い。

#### A-3. 再生中のサンプリングがエディタ更新レートで無制限に走る

`AnimationPlayback.Tick` は `EditorApplication.update` 毎に `SampleAt` + `window.Repaint()` を呼ぶ。エディタ更新は repaint 要求があると毎秒 100〜300 回以上になり得るが、クリップは 60FPS 以下。毎 tick の `SetBlendShapeWeight` × トラック数 + `SyncValuesFromMesh` + `SceneView.RepaintAll` は過剰。

**対策案**: 前回サンプルから 1/FPS 秒未満なら `SampleAt` をスキップ（時刻だけ進める）。

#### A-4. ◆+（全シェイプ挿入）・一括補間変更の O(n²) コスト

`RecordKeys` / `SetAllKeysInterpolation` はループ内で 1 トラックずつ `SetEditorCurve` を呼ぶ。`SetEditorCurve` 1 回のコストがクリップ全体に比例するため、シェイプ 200 個では実質 O(n²) で数秒の停止になり得る。

**対策案**: Unity 2022.1+ の `AnimationUtility.SetEditorCurves(clip, bindings[], curves[])`（複数形）で一括コミットする。

#### A-5. シェイプキーリストの非仮想化描画

`ShapeKeyListUI.DrawList` はスクロール外の行も含め全可視アイテムを GUILayout で毎回描画する。アニメーションモードでは再生・ドラッグで再描画頻度が跳ね上がるため、行数 300 超のアバターでは 1 フレームあたりの Layout+Repaint コストが効いてくる。単体ではフリーズ級にならないが、A-1〜A-3 と複合する。

**対策案**: スクロール位置からの可視範囲カリング（行高は固定なので算出は容易）。

### 3.2 外部要因 — 他ツールとの競合

DenEmo は「SMR ウェイトへの直接書き込み」「クリップへの `SetEditorCurve`」「`SetDirty`」「`Undo.RecordObject`」を高頻度で発行する。これらはすべて `ObjectChangeEvents` / Undo コールバック / インスペクタ再描画を購読している他ツールのリスナーを発火させるため、**同じ操作でも環境によってコストが何倍にも増幅される**。「一部のユーザーだけフリーズ級」の報告は、この増幅で説明できる可能性が高い。

| # | 競合相手 | 想定メカニズム | 切り分け方法 |
|---|---|---|---|
| B-1 | **NDMF プレビューシステム**（Modular Avatar / Avatar Optimizer (AAO) が使用） | NDMF preview はプロキシレンダラーを生成し `ObjectChangeEvents` でシーン変更を監視する。DenEmo の毎イベントの weight 書き込み・`SetDirty` が無効化→再計算を連発させる。特に AAO の Remove Mesh 系プレビューはブレンドシェイプ変更でメッシュ再生成が走り、重量級アバターでは 1 回が秒オーダー | NDMF のプレビューを無効化（Tools → NDM Framework → プレビュー切替）して再現するか確認 |
| B-2 | **Unity 標準 Animation ウィンドウを同じクリップで開いている** | `SetEditorCurve` のたびに Animation ウィンドウ側もリフレッシュ。さらに Animation ウィンドウのプレビュー（AnimationMode）が有効だと、DenEmo の直接書き込みと値を奪い合い、双方が毎フレーム再描画・再サンプルする | Animation ウィンドウを閉じて確認。ドキュメントに「同一クリップを Animation ウィンドウで開いたまま編集しない」を明記 |
| B-3 | **Gesture Manager / Av3Emulator（編集モード動作時）** | エミュレータの Animator が SMR ウェイトを毎フレーム上書き → DenEmo のプレビューと値の取り合いになり、`SyncValuesFromMesh` が毎回違う値を拾って再描画ループが発生 | エミュレーション停止状態で確認 |
| B-4 | **VRCFury** | 編集モードでヒエラルキー/コンポーネント変更を監視するフックを持つ。高頻度の `SetDirty`/Undo 記録がスキャンを誘発する可能性 | VRCFury コンポーネントのないシーンで確認 |
| B-5 | **インスペクタに対象 SMR を表示中** | SMR インスペクタはブレンドシェイプ数分のスライダーを IMGUI 描画する。weight 変更毎に再描画され、200 シェイプではそれ自体が重い | アバターの選択を外して（Inspector をロックせず空にして）確認 |
| B-6 | **重量級シーンビュー（Poiyomi/lilToon + ポスプロ + リアルタイムライト）** | A-1/A-3 の `SceneView.RepaintAll()` 連発のコストがシーン側の描画負荷に比例して増幅 | シーンビューを閉じる/最小化して確認 |
| B-7 | **自動保存系拡張（AutoSave 等）** | `SetDirty` の頻発が保存トリガと噛み合うと `AssetDatabase.SaveAssets` が走り長時間ブロック | 該当拡張の無効化で確認 |

### 3.3 ユーザーへのヒアリング・診断手順（サポート用テンプレート）

1. **環境情報**: アバターのブレンドシェイプ総数（メッシュ毎）、編集中クリップのトラック数・キー数、導入ツール（Modular Avatar / AAO / VRCFury / Gesture Manager / Av3Emulator のバージョン）。
2. **再現操作**: 重いのは (a) スライダードラッグ中 (b) 再生中 (c) キードラッグ中 (d) ◆+ / 一括補間変更 (e) 常時、のどれか。→ (a)(c) なら A-1、(b) なら A-2/A-3、(d) なら A-4、(e) なら B-1/B-3/分離ウィンドウ（A-2）が濃厚。
3. **切り分け**: ①分離タイムラインウィンドウを閉じる → ② Unity の Animation ウィンドウを閉じる → ③ NDMF プレビュー無効化 → ④エミュレータ停止 → ⑤ Inspector の選択解除、の順に 1 つずつ試す。
4. **計測**: Profiler の「Profile Editor」(Editmode) を有効にして再現し、`AnimationUtility.SetEditorCurve` / `Undo.RecordObject` / NDMF 名前空間 / `SceneView.Repaint` のどれが支配的かを確認する。

### 3.4 推奨する修正の優先順位

1. **A-1 のドラッグ中スロットリング + Undo 1 回化 + 部分キャッシュ無効化** — フリーズ報告への直接対処。外部ツールへのイベント発火回数も減るため B 系の増幅も同時に緩和される。
2. **A-2 の常時 Repaint 購読の削除** — 1 行の変更で常時負荷が消える。
3. **A-3 の再生サンプリングの FPS 上限化**。
4. **A-4 の `SetEditorCurves`（複数形）への移行**。
5. **B-2 対策のガード**: DenEmo がクリップ編集中に Unity Animation ウィンドウの AnimationMode がアクティブな場合（`AnimationMode.InAnimationMode()`）、警告表示する。

## 4. まとめ

- **機能面**: 単一キー操作・1 フレームコピペまでは揃っているが、「複数キー選択」「リタイミング」「複数メッシュ対応」「区間ループ再生」が他ツール比の主な穴。特に複数メッシュ非対応はポーズモードとの非対称で、ユーザーが最初に踏む段差になりやすい。
- **UX 面**: 最大の問題は「スライダー操作がクリップに記録される/されない」の 3 状態が暗黙的なこと（2.1）。次いで「全キー補間」の破壊性（2.2）、シーク領域の狭さ（2.3）。いずれも小さな実装で直せる。
- **パフォーマンス面**: フリーズ級の報告は「ドラッグイベント毎のフルコスト記録パイプライン（A-1）」が主犯で、NDMF プレビュー等の変更監視ツール（B-1）が環境依存の増幅器になっているという仮説が最も整合的。スロットリングと部分キャッシュ無効化で内部コストとイベント発火の両方を削るのが最有効。

### 補足（動作確認の観点）

- スライダードラッグの記録は最大 50ms 遅延するが、マウスアップ時に必ず最終値で確定する。ドラッグ中に Undo すると 1 ジェスチャ分がまとめて戻る（Unity 標準と同等の粒度）。
- 分離タイムラインはアイドル時に再描画されなくなったため、もし表示の取り残しがあれば `AnimationModeUI.OnUpdate` の再描画条件を見直すこと。
- `UpdateTrackCurve` は Revision を変更しない。トラックの追加/削除を伴う操作（新規トラックへの初キー・トラック削除・最後のキー削除）は従来どおり `MarkDirty()` で全再構築される。
