# アニメーションモード リファクタリング計画

## 目的
- アニメーションモード関連コードを保守しやすい構成へ再編する。
- 1ファイルあたり約300行を目安に分割し、責務を明確化する。
- SOLID原則に沿って、将来の機能追加時の変更範囲を最小化する。
- 直感的でない仕様・UIを改善し、レイアウトの不統一や崩れを減らす。

## 対象範囲
- `/home/runner/work/DenEmo/DenEmo/UI/AnimationTimelineUI.cs`
- `/home/runner/work/DenEmo/DenEmo/UI/ShapeKeyListUI.cs`
- `/home/runner/work/DenEmo/DenEmo/UI/AnimationModeUI.cs`
- `/home/runner/work/DenEmo/DenEmo/Core/AnimationPreviewController.cs`
- `/home/runner/work/DenEmo/DenEmo/DenEmoWindow.cs`（アニメーションモード連携部分のみ）

## 現状課題（要約）
- `AnimationTimelineUI` に複数責務（再生制御・スクラブ・トラック描画・キー編集）が集中。
- `ShapeKeyListUI` に描画・入力処理・お気に入り・対称編集・スロットリングが混在。
- `DenEmoWindow` がモード遷移、設定保存、UI構築を広く兼務。
- 記号中心ボタン（例: `|◆`, `◆|`, `◇`）の意味が初見で分かりづらい。
- 固定幅・固定高の混在により、言語切替やウィンドウ幅次第で見た目が不安定。

## リファクタリング方針
### 1. 責務分離（SRP）
- タイムラインUIを以下の単位へ分割する。
  - 再生コントロール描画
  - ルーラー/スクラバー描画
  - トラック一覧描画
  - キーフレーム操作メニュー
- シェイプキー行描画を「単体行」「対称統合行」「共通入力処理」に分離する。
- モード遷移・設定永続化・表示ロジックを `DenEmoWindow` から段階的に切り出す。

### 2. 拡張性向上（OCP/DIP）
- 補間モードの扱いをUI実装から疎結合化し、追加時の変更点を限定する。
- キーフレーム操作（追加/削除/補間変更）をUIイベントから分離し、操作APIを一本化する。

### 3. 重複削減
- スライダー変更時の共通処理（Undo・ドラッグ中反映・確定反映）を共通化する。
- キーフレーム有無判定・トグル操作の分岐を集約する。

### 4. UI/UX改善
- 記号ボタンにツールチップを必須化し、操作意図を明示する。
- Trackフィルタの文言を「表示対象が何か」まで含めて明確化する。
- 破壊的操作（トラック削除等）は一貫した確認導線に揃える。

### 5. レイアウト改善
- 幅/高さの定数を統合し、ボタンサイズの規格を統一する。
- 固定幅依存を減らし、可変幅レイアウトを優先する。
- 2言語表示時のラベル長に耐える最小幅/折返し方針を定義する。

## ファイル分割の目安（300行前後）
- `AnimationTimelineUI.cs` を 2〜4 ファイルへ分割し、各ファイルを 300行以下に維持。
- `ShapeKeyListUI.cs` を 2〜3 ファイルへ分割し、行描画と入力処理を分離。
- `DenEmoWindow.cs` はアニメーションモード関連ヘルパーを分離し、責務境界を明確化。
- 共通ヘルパーは `UI/` または `Core/` に配置し、再利用可能な単位で定義。

## 実施ステップ
1. 依存関係を維持したままファイル分割（振る舞い不変）。
2. 共通処理抽出（スライダー入力・キーフレーム操作）。
3. UI文言/ツールチップの整備。
4. レイアウト定数の統合とサイズ統一。
5. モード境界の整理（`DenEmoWindow` の責務縮小）。
6. 回帰確認（既存操作フロー: 再生、REC、キー追加/削除、保存、フィルタ）。

- アニメーションモード画面で明確なレイアウト不統一が解消されている。
